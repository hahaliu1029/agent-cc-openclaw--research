一、核心原则：上下文窗口是第一约束

"Most best practices for Claude Code are based on one constraint: Claude's context window fills up fast, and performance degrades as it fills."
— Anthropic Best Practices

Claude Code 的优化策略都围绕一个事实：上下文窗口是有限资源。会话上下文包含对话历史、已读文件、命令输出、CLAUDE.md、Skills 与系统指令等；随着上下文增大，Claude Code 会自动管理（先清理旧工具输出，必要时总结对话），但主动管理上下文仍然是降低成本和保持质量的关键。

1.1 上下文消耗来源

来源 消耗量 加载时机
CLAUDE.md + rules 中 会话启动时
对话历史 高（持续增长） 每轮累积
文件读取结果 高 每次 Read/Grep
工具调用输出 中-高 每次工具使用
MCP 工具定义 低-中 延迟加载或启动时
Skills 低 按需加载

1.2 上下文管理策略

主动压缩（最重要的习惯）：

# 在逻辑阶段边界处手动压缩
/compact 保留 API 变更和数据库 schema 的上下文，总结之前的调试步骤

# 切换到不相关任务前
/compact 保留当前进度
/clear

# 也可以在 CLAUDE.md 中指定压缩策略
# Compact instructions
# When using compact, focus on:
# - Test failures and error messages
# - Code changes made
# - Key architectural decisions

上下文退化阈值（社区测试数据）：

使用率 表现影响 建议操作
低占用 无明显退化 正常工作
中等占用 精度开始下降，早期指令可能被压缩 考虑 /compact
高占用 幻觉增加，指令遵从率下降 立即 /compact
接近上限 严重退化，自动压缩触发 强制 /clear 重新开始
注：官方未公开固定百分比阈值。可通过 /context 命令观察当前占用情况，建议在阶段性任务完成后主动 /compact，而非等待自动压缩。

为什么手动压缩优于自动压缩：

  • 自动压缩在接近上下文上限时触发（官方未公开固定阈值）— 此时你已经为所有之前的轮次支付了高额 token 成本
  • 手动压缩让你控制什么被保留
  • 在逻辑断点压缩保留关键推理上下文

在 CLAUDE.md 中指定压缩策略（推荐）：

## Compact Instructions
完成阶段性任务后建议 /compact，压缩时保留：代码变更摘要、未完成的待办、关键架构决策。
切换到不相关任务前先 /clear。

用子 Agent 保护主上下文：

# BAD：在主会话中探索大量文件
读取 src/ 下所有文件来理解架构

# GOOD：委托给子 agent
使用一个子 agent 来调研认证模块如何处理 token 刷新

子 agent 在独立的上下文窗口中运行。它们的详细输出（文件读取、探索过程）不会污染主对话，只返回摘要。

1.3 关键命令

命令 作用 何时使用
/compact [focus] 压缩对话历史，可指定保留重点 每个逻辑阶段完成后
/clear 重置上下文 切换到不相关任务时
/context 查看上下文使用情况 监控上下文消耗
/cost 查看 token 消耗和费用 追踪成本
claude --continue 恢复最近的会话 继续之前的工作
claude --resume 从历史会话中选择恢复 回到特定任务

二、CLAUDE.md —— 项目记忆文件

2.1 什么是 CLAUDE.md

CLAUDE.md 是 Claude Code 的持久记忆文件。每次会话启动时自动加载，其内容作为系统上下文的一部分影响 Claude 的所有行为。类似于 Cursor 的 .cursorrules 或 GitHub Copilot 的 .github/copilot-instructions.md。

2.2 文件层级与加载顺序

~/.claude/CLAUDE.md                    # 个人级别，所有项目生效
~/.claude/rules/*.md                   # 个人规则文件，按需加载
<project-root>/CLAUDE.md              # 项目级别，通过 git 共享
<project-root>/.claude/rules/*.md     # 项目规则文件，按需加载
<subdirectory>/CLAUDE.md              # 子目录级别，仅在该目录下生效

所有匹配的文件会叠加而非覆盖。更具体的文件在更通用的文件之上增加指令。

**特殊文件：CLAUDE.local.md**

在项目根目录创建的 CLAUDE.local.md 会被 Claude 自动读取，且自动被 gitignore。适合存放：

  • 个人本地后端地址（不同开发者不同）
  • 个人调试偏好
  • 临时的工作备忘

2.3 该写什么 vs 不该写什么

该写：

# CLAUDE.md 模板

## 项目概览
[1-3 句话：是什么、技术栈、主语言]

## 架构
[目录结构，每个顶层目录的简要描述]
[关键模式：分层架构、DDD、微服务等]

## 常用命令
npm install              # 安装依赖
npm run dev              # 启动开发服务器
npm test                 # 运行全部测试
npm test -- --grep "name" # 运行单个测试（重要！Claude 会频繁使用）
npm run lint             # 代码检查
npm run build            # 构建

## 代码规范
- [Commit 格式]
- [分支策略]
- [代码风格规则]
- [框架特定模式]

## 关键约束
- [绝不能违反的硬性规则]
- [安全边界]
- [导入限制]

## CI/CD
[流水线检查哪些内容]

不该写：

不该写的内容 原因 替代方案
密钥、API Key、Token 安全风险 使用环境变量
大段代码或文档 消耗上下文 使用 @docs/api.md 引用
频繁变化的信息 容易过时 让 Claude 按需查询
工具权限配置 不属于自然语言指令 使用 settings.json
README 的重复内容 浪费空间 CLAUDE.md 是给 Claude 的，不是给人的
显而易见的事 "写干净的代码"没有信息量 写具体的、非显而易见的约束

2.4 保持精简

目标：200 行以内。 CLAUDE.md 在每次会话启动时加载，消耗 token。

当内容超过 200 行时：

  • 将专题内容拆分到 .claude/rules/*.md（按需加载）
  • 将可复用工作流移到 .claude/commands/*.md（按需调用）
  • 将专业知识移到 .claude/skills/（标准位置，Claude Code 自动发现）

2.4.1 Path-Scoped Rules（路径作用域规则）

.claude/rules/*.md 支持 YAML frontmatter 中的 paths: 字段，使规则仅在 Claude 操作匹配路径的文件时加载：

---
paths:
  - "src/route/**"
  - "src/views/Monitoring/**"
---

# 三级菜单 Tab 路由规约
...

效果：当 Claude 只在 src/api/ 下工作时，路由规约不会被加载，节省上下文。

适用场景：

  • 模块专属规约（如路由、监控布局）→ 加 paths 限制
  • 通用规约（如代码质量清单）→ 不加 paths，始终可用

2.4.2 @ 引用语法

CLAUDE.md 中可以用 @path/to/file 语法引用其他文件（支持最多 5 层递归），Claude 会在需要时自动拉取内容：

# 当前写法
详细规约见 `.claude/rules/routing-conventions.md`。

# @ 引用写法（Claude 会按需拉取内容）
详细规约见 @.claude/rules/routing-conventions.md。

注意：@ 引用会将文件内容嵌入上下文。对于已通过 .claude/rules/ 自动加载的文件，不需要再用 @ 引用。@ 更适合引用不在 .claude/ 目录下的文件，如 @ui-guideline.md。

2.5 真实世界示例

Actus 项目（~124 行，结构良好）：

  • Project Overview → Architecture → Agent Orchestration → LLM Adapters → Tool System → Skill Creator → Common Commands → CI Pipeline → Configuration → Conventions
  • 每个部分都是 Claude 工作时需要的具体可操作信息
  • 末尾有行为指令：注意：不要提交代码，永远留给我提交

本项目（snap-frontend，199 行，五层架构）：

重构前 ~425 行，重构后 199 行。采用五层配置架构：

CLAUDE.md (199行，每次加载)
├── Hard Constraints、Project Snapshot、Architecture、Theme、API Planning
├── On-Demand Rules 索引、Custom Commands 索引、Skill 入口
│
├── .claude/rules/ (按需加载)
│   ├── code-quality.md (98行) — 代码质量检查清单、ESLint、自检流程
│   ├── capabilities-reference.md (119行) — 能力参考、图标/表格/Antd文档源
│   └── routing-conventions.md (41行) — 三级菜单Tab路由规约 [paths: src/route/**, src/views/Monitoring/**]
│
├── .claude/commands/ (按需调用)
│   ├── lint.md — /lint [files]
│   ├── self-check.md — /self-check [files]（引用 /lint + code-quality.md）
│   ├── new-page.md — /new-page <name>
│   └── new-api.md — /new-api <feature>
│
├── .claude/skills/ (自动发现)
│   ├── snap-front-end-encyclopedia/ — 全项目通用能力百科
│   └── monitoring-page-layout/ — Monitoring模块布局标准
│
└── .claude/settings.local.json — 权限 + Hooks

关键经验：

  • 代码质量清单（~150 行）是最大的拆分收益项
  • /self-check 命令引用 /lint 和 code-quality.md，避免三处维护同一逻辑
  • Skills 放在 .claude/skills/ 而非 docs/skills/，利用 Claude Code 自动发现机制
  • rules 文件用 paths: frontmatter 限制加载范围

2.6 配置层级总览

关注点 放在哪里
工具权限、MCP 服务器、Hooks settings.json / settings.local.json
项目架构、规范、构建命令 CLAUDE.md（项目级，≤200 行）
模块专属详细规约 .claude/rules/*.md（按需加载，可加 paths 限制）
可复用 prompt 模板 .claude/commands/*.md（按需调用）
领域知识与工作流 .claude/skills/（自动发现，按需激活）
个人偏好（语言、风格） ~/.claude/CLAUDE.md 或 ~/.claude/rules/*.md
个人本地配置 CLAUDE.local.md（自动 gitignore）
安全策略、commit 规则 ~/.claude/rules/*.md（个人级）

三、高效提示技巧

3.1 具体优于模糊

质量 示例
太模糊 "修复登录 bug"
较好 "用户报告会话超时后登录失败。检查 src/auth/ 中的 token 刷新"
最佳 "会话超时后登录失败。检查 src/auth/tokenRefresh.ts。先写一个失败的测试，然后修复它。用 npm test 验证"

3.2 高效提示的模式

1. 提供验证标准：

写 validateEmail 函数。
测试用例: "user@example.com" → true, "invalid" → false, "user@.com" → false。
实现后运行测试。

2. 引用具体文件：

修复认证流程。问题在 src/auth/session.ts 的 token 刷新逻辑中。

3. 描述症状而非解决方案：

# GOOD：描述症状
用户删除旧文件时报 500 错误。Error: "EACCES: permission denied"。检查 src/fileStore.ts

# BAD：直接给方案
修改文件权限

4. 使用 @ 引用文件：

修复 @src/auth.ts 中的 token 过期处理

5. 使用图片/截图：

[粘贴 UI 截图]
实现这个设计。完成后截图对比原图，列出差异并修复。

3.3 纠正两次后重新开始

当你纠正 Claude 两次但问题仍存在时，上下文已被失败尝试污染。正确做法：

/clear
# 用一个更好的初始 prompt 重新开始，把你从之前的尝试中学到的信息融入进去

四、工具使用指南

4.1 选对工具

任务 最佳工具 避免
按名称查找文件 Glob find, ls
搜索文件内容 Grep grep, rg
读取已知文件 Read cat, head, tail
修改已有文件 Edit sed, awk
创建新文件 Write echo >, cat <<EOF
执行命令 Bash —
复杂多步研究 Agent（子 agent） 在主上下文中逐个探索

4.2 Read：精确读取

# GOOD：只读需要的部分
Read("src/auth.ts", offset=50, limit=30)  # 只读第 50-80 行

# BAD：读取整个大文件
Read("src/auth.ts")  # 10000 行文件全部进入上下文

4.3 Grep → Read 工作流

# 第一步：用 Grep 定位
Grep("handleTokenRefresh", "src/**/*.ts", output_mode="files_with_matches")

# 第二步：只 Read 相关文件
Read("src/auth/tokenRefresh.ts")

4.4 Edit vs Write

  • Edit：只发送差异，上下文开销小，更安全
  • Write：完全覆盖文件，仅用于新建文件或完全重写
  • 规则：必须先 Read 再 Edit/Write

4.5 Bash：最小化输出

# BAD：返回 10 万行
npm test

# GOOD：过滤输出
npm test 2>&1 | grep -E "FAIL|PASS|Error" | head -20

五、工作流模式

5.1 Plan Mode（复杂任务必用）

当面对多文件变更、不熟悉的代码库或大规模重构时：

Shift+Tab  → 进入 Plan Mode

工作流：

  1. 探索（只读）：Claude 分析请求，读取相关文件
  2. 规划：Claude 产出结构化计划（涉及哪些文件、修改顺序、风险点）
  3. 迭代：你审查计划，提出修改（"加上错误处理" / "跳过步骤 3"）
  4. 执行：切回正常模式，Claude 按计划逐步实施

5.2 TDD 工作流

1. "为 validateEmail 写失败的测试。不要实现。"
   → Claude 产出测试文件（RED）

2. "实现最小代码让测试通过。"
   → Claude 实现功能（GREEN）

3. "重构，保持测试通过。"
   → Claude 优化代码（IMPROVE）

4. "运行 npm test 并检查覆盖率。"
   → 验证 80%+ 覆盖

5.3 调试工作流

1. 粘贴完整的错误堆栈
2. "先写一个复现这个 bug 的失败测试"
3. "生成 3 个可能的原因假设，逐一验证"
4. "修复并确认测试通过"

关键原则： 先复现，后修复。不要直接"修 bug"，先证明 bug 存在。

5.4 多 Agent 并行

# 三个独立任务，用并行子 agent
对认证模块做安全审查、对缓存层做性能审查、对工具函数做类型检查，
使用三个独立的子 agent 并行执行。

模型路由：

  • 轻量级 agent（搜索、格式化、测试运行）→ Haiku 4.5（成本约为 Sonnet 的 1/3）
  • 核心开发 agent（代码生成、重构）→ Sonnet 4.6
  • 架构决策 agent（复杂推理）→ Opus 4.6

5.5 Git 集成

# 让 Claude 从 diff 生成 commit message（比从记忆生成更准确）
git diff --staged | claude -p "基于这个 diff 生成 conventional commit message"

# PR 描述
git diff main...HEAD | claude -p "为这个 PR 写详细的描述和测试计划"

5.6 Writer/Reviewer 双会话模式

# 会话 A：写代码
实现 API 端点的限流中间件

# 会话 B（新开）：以新视角审查
审查 @src/middleware/rateLimiter.ts，检查边界条件、竞态条件、一致性问题

新会话没有写代码时的上下文偏见，审查更客观。

六、配置详解

6.1 settings.json 结构

{
  // 权限管理
  "permissions": {
    "allow": [
      "Read",                    // 所有文件读取
      "Glob",                    // 文件搜索
      "Grep",                    // 内容搜索
      "Bash(git *)",             // 所有 git 命令
      "Bash(npm test)",          // npm test
      "Bash(npm run *)",         // 所有 npm scripts
      "Edit",                    // 文件编辑
      "Write"                    // 文件写入
    ],
    "deny": [
      "Bash(rm -rf *)",         // 阻止危险删除
      "Bash(sudo *)",           // 阻止 sudo
      "Bash(npm publish*)",     // 阻止意外发布
      "Bash(git push --force*)", // 阻止强制推送
      "Bash(git reset --hard *)" // 阻止丢失改动
    ]
  },

  // 模型配置
  "model": "claude-sonnet-4-6",

  // 注意：MCP 服务器配置在 .mcp.json 中，不在 settings.json

  // Hooks
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write|Edit",
      "hooks": [{ "type": "command", "command": "prettier --write $CLAUDE_FILE_PATH 2>/dev/null || true" }]
    }]
  },

  // 行为
  "alwaysThinkingEnabled": true
}

6.2 配置作用域

Enterprise 管理策略       ← 最高优先级（不可覆盖）
    ↓
~/.claude/settings.json  ← 个人全局
    ↓
.claude/settings.json    ← 项目级（提交到 git）
    ↓
.claude/settings.local.json ← 项目本地（gitignore）

6.3 权限模式

模式 自动批准 切换方式
default 仅文件读取 Shift+Tab 循环
acceptEdits 读取 + 编辑 迭代开发时
plan 仅读取（只读） 探索、规划时
auto 一切（分类器检查） 长任务，信任方向
bypassPermissions 一切（无检查） 仅限隔离容器

6.4 Hooks 系统

Hook 类型：

Hook 触发时机 能否阻止操作
PreToolUse 工具执行前 是（exit 2）
PostToolUse 工具执行后 否
Notification 通知发出时 否
UserPromptSubmit 用户提交输入时 是
Stop Claude 完成回复时 否

实用 Hook 示例：

# 写入后自动格式化
#!/bin/bash
# ~/.claude/hooks/post-edit-format.sh
FILE="$CLAUDE_FILE_PATH"
if [[ "$FILE" == *.ts || "$FILE" == *.tsx ]]; then
  npx prettier --write "$FILE" 2>/dev/null
fi
# 写入后自动 ESLint 检查（本项目实战示例）
# settings.local.json hooks 配置
# 当 Write 或 Edit 工具执行后，自动对 .js/.jsx/.ts/.tsx 文件运行 ESLint
# matcher: "Write|Edit"
# command: 从 stdin JSON 提取 file_path，如果是 JS/TS 文件则执行 eslint
# 阻止写入包含密钥的内容
#!/bin/bash
# ~/.claude/hooks/validate-no-secrets.sh
INPUT=$(cat -)
CONTENT=$(echo "$INPUT" | jq -r '.content // empty')
if echo "$CONTENT" | grep -qiE '(api_key|secret|password)\s*=\s*["\x27][^\s]+'; then
  echo "BLOCKED: 检测到硬编码密钥。请使用环境变量。"
  exit 1
fi

Hook 输入输出机制：

Hook 通过 stdin 接收结构化 JSON 数据（包含工具名、参数、输出等），而非环境变量。可用的环境变量仅限于项目/插件级别：

变量 说明
CLAUDE_PROJECT_DIR 项目根目录
CLAUDE_PLUGIN_ROOT 插件根目录
CLAUDE_PLUGIN_DATA 插件数据目录
CLAUDE_ENV_FILE 环境文件路径

stdin JSON 示例（PreToolUse）：

{
  "tool_name": "Edit",
  "tool_input": { "file_path": "/src/auth.ts", "old_string": "...", "new_string": "..." }
}

社区高频推荐 Hooks：

Hook 类型 用途 优先级
PostToolUse `Write\ Edit` → ESLint 写入后自动 lint 检查 P0
PostToolUse `Write\ Edit` → Prettier 写入后自动格式化 P1
PreToolUse `Write\ Edit` → 密钥检测 拦截含 sk-*/AKIA*/ghp_* 等模式的写入 P0
PreToolUse Bash → 危险命令拦截 阻止 rm -rf、git push --force P1
Stop → 完成度验证 Claude 完成回复时检查是否真正完成 P2

Hook 性能建议：

  • 安全拦截（PreToolUse）必须同步执行
  • 格式化、日志、测试类 hooks 建议设置 timeout 避免阻塞
  • exit 0 = 成功，exit 2 = 阻塞性错误（对支持阻止的事件生效，如 PreToolUse 阻止工具调用、UserPromptSubmit 拒绝输入），其他非零 = 非阻塞性错误（执行继续，stderr 仅 verbose 模式可见）
来源：Claude Code Hooks 官方文档、Trail of Bits claude-code-config

6.7 Skills（.claude/skills/）

Skills 是比 commands 更重的知识载体，支持自动发现和丰富的 frontmatter 配置。

Skills vs Commands vs Rules 对比：

维度 Rules Commands Skills
加载方式 自动（可加 paths 限制） 手动 /命令名 元数据始终在上下文，SKILL.md 按需加载
触发者 始终生效 仅用户 用户 + Claude 自动（可配置）
内容类型 规约条文 可执行工作流 领域知识 + Playbook + 可附带脚本/参考文件

上下文成本 高（始终占用） 低（仅调用时） 最优（渐进披露）
适合场景 代码质量、命名规范 lint、自检、脚手架 组件百科、布局标准、调试方法论

Skill 目录结构（官方示例 + 推荐扩展）：

my-skill/
├── SKILL.md              # [必须] 主指令文件（<500 行，<5k token）
├── scripts/              # [可选] 可执行脚本（Claude 执行而非读取）
│   └── validate.sh
├── examples/             # [可选] 使用示例（Claude 按需读取）
│   └── sample.md
├── reference.md          # [可选] 详细 API 参考文档
└── template.md           # [可选] 模板文件（Claude 填充）

渐进披露（Progressive Disclosure）三级加载：

  1. 元数据（~100 词）：name + description，始终在上下文
  2. SKILL.md 正文（<500 行）：skill 被触发时加载
  3. 子文件（无限）：SKILL.md 中引用，按需加载

Frontmatter 关键配置建议：

  • description 应前置核心触发场景和关键词，Claude 根据 description 决定是否加载 skill（超过 250 字符会被截断）
  • disable-model-invocation: true：有副作用的操作（deploy、commit）禁止自动触发
  • allowed-tools：只读分析用 Read,Grep,Glob；文件修改用 Read,Edit,Write

十、常见陷阱与解决方案

陷阱 1：万能会话

问题： 一个任务，然后不相关的问题，再回到第一个任务。上下文充满无关信息。

解决：

/rename oauth-migration
/clear
# 处理第二个任务
claude --resume   # 选择 "oauth-migration" 回到第一个任务

陷阱 2：不读就改

问题： 直接编辑未读过的文件，导致上下文缺失或修改错误。

解决： Claude Code 强制要求先 Read 再 Edit/Write。养成 Grep → Read → Edit 的习惯。

陷阱 3：CLAUDE.md 过长

问题： 超过 200 行的 CLAUDE.md，重要规则被淹没。

解决： 拆分到 .claude/rules/*.md（按需加载）和 .claude/commands/*.md（按需调用）。

陷阱 4：无限探索

问题： "调查一下这个问题"没有范围限制。Claude 读取数百个文件，上下文耗尽。

解决：

# 限定范围
使用子 agent 调研 src/auth/ 中的 token 刷新

# 或者
调查：哪些文件负责会话验证？

陷阱 5：信任但不验证

问题： Claude 产出看起来合理但未处理边界情况的代码。

解决： 总是提供验证标准：

写 validateEmail。测试: "user@example.com" → true, "invalid" → false。
实现后运行测试。

陷阱 6：权限过宽

问题： Bash(*) 允许一切，可能造成意外损坏。

解决： 使用最具体的权限模式：

{
  "permissions": {
    "allow": ["Bash(npm test)", "Bash(git commit *)"],
    "deny": ["Bash(rm -rf *)", "Bash(sudo *)"]
  }
}

十一、复合工程（Compounding Engineering）

每次 Claude 犯错时，将纠正写入 CLAUDE.md 或 rules，使其成为永久上下文，确保同类错误不再重复。

11.1 核心理念

Claude Code 的 CLAUDE.md 是一个可累积的知识库。每次会话中发现的项目特定陷阱，都应该被提炼为规则持久化下来。

11.2 实践方法

1. 发现 Claude 犯了一个项目特定的错误（如使用了废弃 API、选错了 CSS 基线）
2. 纠正 Claude 并验证修复
3. 将纠正提炼为规则：
   - 高频通用 → 写入 CLAUDE.md Hard Constraints
   - 模块专属 → 写入 .claude/rules/<module>.md
   - 代码质量 → 追加到 .claude/rules/code-quality.md
4. 下次会话中 Claude 自动遵循，不再重复犯错

11.3 本项目示例

本项目的 code-quality.md 就是复合工程的产物 — 从反复出现的 SonarQube 扫描问题和代码审查反馈中提炼出的 40+ 条强制规则。每发现一个新的高频问题模式，就追加一条规则。

十二、效能层级总结

按经验影响排序（具体比例因项目和使用方式而异，可通过 /cost 验证）：

1. 上下文管理    (影响最大)  ← 频繁 /clear 和 /compact，使用子 agent
2. 精确提示      (影响较大)  ← 引用文件，给出示例，描述症状
3. 验证驱动      (影响较大)  ← 测试、截图、期望输出
4. 配置优化      (影响中等)  ← Hooks、权限、Skills
5. 工具选择      (影响中等)  ← Grep vs Read, Edit vs Write

先掌握 1-3，再优化 4-5。 基础扎实后再追求进阶。
